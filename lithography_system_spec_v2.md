# 光刻工艺互动展示系统 - 重构需求与技术规格书 (V2.0)

## 1. 项目概述

本项目旨在重构原有的光刻效果展示网页。核心改造目标：彻底废弃笨重的视频（MP4）叠加方案，采用"纯图片分层 + WebGL 实时算法"实现 AE 级别的色光与材质渲染；同时，从零搭建一个基于 Web 的动态管理后台（CMS），实现项目资产的云端管理、动态调参和对客交付。

## 2. 系统架构与交互流程

系统分为前端用户展示层 (C端) 与 后台管理层 (B端)。

### 2.1 前端双入口设计 (Home)

首页提供两个核心分支入口：

- **公域展示（工艺陈列室）**：
  - 目标：向潜在客户展示公司落地实力，同时提供基础工艺库供销售展示。
  - 核心交互：无需登录。点击进入后，页面顶部提供极简 Tab 切换：【商业案例 (Cases)】与【工艺图谱 (Textures)】。
  - 视觉形态 (双模式自由切换)：页面右上角常驻**"视图模式切换器 (View Toggles)"**：
    - **模式一 (3D Carousel)**：沉浸式的 3D 轮播阵列，供客户"随便翻一翻"，感受光刻特效的空间美感。
    - **模式二 (Grid Matrix)**：精密网格平铺阵列，卡片间距保持适度（不过挤也不过空，兼顾设计感与实用性），鼠标悬浮时附带简洁的浮动微动画。
      - **高级检索 (Craft Search)**：顶部提供全局搜索框，支持按工艺名称、介绍进行关键词筛选。
      - **详情弹出胶囊 (Capsule Detail)**：点击工艺卡片后，主视图变灰并模糊，弹出沉浸式“胶囊”弹窗。展示工艺大图、详细介绍、推荐尺寸及优缺点分析。支持点击遮罩或关闭按钮快速退出。
- **私域交付（客户专属通道）**：
  - 交互：点击后弹出极简登录框。客户输入管理员提供的 6位提取码（或使用"记住设备"功能实现 7 天免密），验证通过后，进入该客户专属的项目列表页。

### 2.2 核心展示页 (Index - WebGL 渲染引擎)

这是客户查看效果的核心互动页。

- **启动交互**：
  - **移动端**：进入页面后，展示一个朴实但有设计感的"开始预览"按钮，用于获取浏览器陀螺仪权限。
  - **PC 端**：无需额外按钮，页面加载完成即自动进入鼠标控制模式。
- **无侵入式体验牵引 (Onboarding UX - 零操作防呆)**：为防止客户因非专业出身而产生“不知道怎么看”的认知断层，系统强制嵌入以下视觉暗示（**所有面向客户的文案均为中文**）：
  - **引诱转动 (Auto-Tease)**：加载完毕的前 3 秒，若用户无任何输入，方盒会自动进行带缓动曲线的微幅“呼吸自转”，并伴随半透明线框小手动画提示 `[滑动或转动设备查看细节]`。一旦用户接管输入，提示瞬间消散。
  - **对比呼吸灯 (Hold-to-Compare Affordance)**：右下角的版本对比按钮不仅常驻，更带有持续的极光扫过微动效（Pulsing Sweep）抓取余光。若系统检测到用户游览超过 10 秒仍未点击过比对，按钮旁会自动弹出轻气泡引导 `[长按此处查看上一版]` 随后淡出。
  - **微型坐标系 (Axis Compass)**：屏幕一角提供极其袖珍的、与主视角绝对同步的 3D 线框视角器（标识 **Front / Top / Left** 等英文朝向，更显工业高级感），防止用户迷失空间感。此组件用纯 CSS 3D Transform 实现，不占用 WebGL 主渲染管线。
  - **副作用防御条款**：
    - **GPU 卸载 (Performance Guard)**：所有引导动效（自转、小手动画、气泡）在用户首次交互后必须从渲染循环中彻底移除（非隐藏），释放 GPU 资源给主场景。
    - **触摸冲突隔离 (Touch Isolation)**：右下角对比按钮设置 150ms 长按判定延迟，与主区域的“滑动旋转”事件物理隔离，避免误触。
    - **微信滑动防穿透 (Overscroll Isolation)**：针对微信/iOS常见的“下拉回弹黑底”与“滑出页面”痛点，强制在 3D 交互区接管所有 touch 事件 (`touch-action: none`)，确保用户的滑动 100% 作用于盒子，防止误退浏览器。
    - **引导去重 (Onboarding Dedup)**：利用 localStorage 记录“已引导过”状态。同一设备只在首次访问时显示完整引导，后续访问不再弹出。
- **交互控制**：支持手机端陀螺仪（重力感应）与 PC 端鼠标滑动，实时改变光影和图层视差。
- **神级历史对比交互 (Hold-to-Compare)**：针对客户存在多版本的状态，采用高级的“无缝穿梭”策略。用户将其转选到一个极佳的看光角度后，按住右下角的 `[查看版本 V1]`小圆角悬浮按钮（带缓慢呼吸扫光）。此时 3D 盒子的所有空间矩阵冻结继承，底图瞬间替换。松开恢复当前版本。（注：若无历史版本，该按钮自动彻底隐藏，不给客户视觉噪音）。
- **辅助平面图模式 (Unfolded View)**：默认进入 3D 模式。在屏幕底部居中提供类似 iOS 胶囊分段器的 `[立体 | 展开]` 按钮。点击展开时，3D 盒子带动画淡出，六面平面十字图淡入（0.5秒 Crossfade），用于确认花纹铺贴位置。进入平面模式时，右下角的对比按钮自动隐藏。
- **视觉与美学基调 (Aesthetic Directives)**：
  - **加载动画 (Loading)**：摒弃传统圆圈，采用极简“细光线进度条” + 现代非衬线字体数字。加载完毕后光线从中间向两侧丝滑拉开。
  - **沉浸式背景**：底层采用暗调径向渐变（深灰蓝 → 纯黑），上层叠加极淡的科技点阵（opacity: 0.03）。
  - **视差微动 (Parallax Panning)**：用户转动 3D 盒子时，背景层沿反方向产生幅度极小的位移（如 1/10 比例），营造脱离感与空间纵深，用户停止操作则背景完全静止，零开销。
- **纯粹单向展示**：页面不设任何“确认/驳回”等审批流操作按钮，确保 100% 页面空间留给视觉，商业链路流转回归即时通讯工具。
- **一键纯净截图 (One-Click Screenshot)**：提供快捷截图按钮，隐藏所有前端 UI 控制栏，输出最高精度的当前 3D 视角图片，便于客户单据留存与二次分享。

### 2.3 Web 管理后台 (Admin)

管理员使用的线上 CMS 控制台（取代线下软件）。

- **项目管理**：新建项目，生成提取码，设置项目名称、描述。
- **渲染类型配置**：选择模型类型（方盒 Box 或 圆柱 Cylinder），类型决定前端调用哪种渲染逻辑。
- **分层资产管理 (方盒模式深度定制)**：
  - **智能收纳面板与手风琴 UI**：
    - **小白友好默认展开**：首次进入项目编辑页时，所有面板默认展开，左上角提供明确的文字按钮 `[← 收起配置台]`。
    - **极简折叠**：点击收起后，面板缩减为 48px 宽的竖条，悬停时可预览。展开收起状态由浏览器 localStorage 记忆。
    - **互斥叠罗汉 (Accordion)**：展开某一图层的参数面板时，其他层自动折叠收起。
  - **动态图层栈与重命名**：面上可无限新建图层（通过统一的加号），支持双击重命名（完全支持中文）。
  - **面属性复用与隐藏**：针对相同图案，提供"同步至某面"的功能；支持一键开启/关闭特定面的渲染。
  - **层级角色与专属参数**：
    - **静态图 (Static)**：提供"作为蒙版遮罩 (Mask)"复选框。
    - **云模图 (Cloud)**：提供"视差强度 (Parallax Scale)"工业级滑块（代替物理深度）。
    - **灰度图 (Specular)**：定义反光通道，无额外参数。
  - **全局光影大层**：除单面配置外，支持在最外层挂载包裹整个方盒的"全局光影图层"。
  - **全局效果模板 (Effect Templates)**：基于“一切图层皆可拔插”的理念，管理员可将调校好的一整套方盒图层架构（底图 + 特定视差的云膜 + 灰度高光分布等）整体保存为【模板】。后续新建相似产品时可一键套用"配方"再局部微调。
- **多版本生命周期与状态归档 (Version Snapshots & Archiving)**：
  - 具备"草稿 / 正式发布"隔离机制，防止调试过程暴露给客户。
  - **基于克隆的迭代与备注**：针对客户反复修改的痛点，系统支持在发布版上**“创建新版本 (Duplicate)”**。克隆时必须填写**“版本更新备注”**（如：*加深了正面云层质感*）。原有的 `v1` 被冻结。
  - **对客交付级比对**：管理员在 `v2` 上增量编辑发布后，客户访问链接时，结合 2.2 节的 **“Hold-to-Compare”长按无缝对比**，客户可以在同一个 3D 空间下，极速比对新旧版本的差异（菜单会清晰展示由管理员填写的版本更新备注）。
  - **结案归档与资产瘦身 (Project Archiving)**：当项目定稿投产后，管理员执行“完结归档”操作。系统将执行两大逻辑：① 仅保留客户最终确认的【决定版】配置，冻结并清除其他所有中间废弃版本；② **垃圾回收器 (Garbage Collector)** 启动，自动从云端存储删除仅存在于废弃版本且最终版未使用的多余图片资产，释放资源。归档项目转为只读历史，对外提取码彻底失效。

## 3. 核心技术规范与"避坑"指南（重点干货）

AI 助手在编写代码时，必须严格遵守以下技术选型与规范，避免走弯路：

### 3.1 渲染引擎规范 (告别 CSS 堆叠)

- **禁用方案**：严禁使用纯 CSS mix-blend-mode 来模拟光效，质感无法达到商业要求。
- **指定方案**：必须使用 WebGL (推荐 Three.js 或 Pixi.js) 编写自定义着色器（Shader）。
- **分层渲染逻辑**：
  - **静态图 (Static Layer)**：作为底层或顶层。激活"蒙版"属性时，提取非透明区域作为全局遮罩（Mask），在之上开"窗口"。
  - **云模图 (Parallax Cloud Layer) - [算法优化]**：**废弃物理 Z 轴真实位移**避免穿模风险。采用 **纯 UV 视差映射算法 (UV Parallax Mapping)**。云膜在 3D 空间上与底图绝对贴合，通过获取视角向量动态偏移内部图像纹理坐标，完美制造"透过蒙版窗口看到底下深层空间"的错觉。**视差位移量需基于物理真实感进行参数校准，确保在对应观看距离下给人的深度感受与现实基本一致。**
  - **灰度图 (Specular/Height Map)**：作为 Alpha 通道输入 GPU，仅在偏白区域产生强烈高光反射。
- **异形渲染逻辑**：
  - **方盒模式**：执行基于平面的 3D 透视倾斜计算，6 个视图基于相对位置拼合成真实 3D 盒子。**方盒尺寸与防破模策略：**
    - 尺寸自动推导：默认根据正视图宽高比确定本体前后面尺寸，结合左/右侧视图宽高比推断厚度。
    - **柔性防呆机制 (容错拉伸 & 警告)**：为避免由于上传比例错误导致 3D 盒子在边缘处裂开，系统强制以正视图的高度作为全局“Y 轴硬标准”。当侧面图的高度像素不能匹配正视图时：
      - **微小差距**：系统悄无声息地进行微小裁剪或拉伸（自适应适配），确保严丝合缝闭环不漏光。
      - **差距过大**：下方红字警示（如：“侧面图比例与正面图差异较大，建议核对”），但依然允许管理员强行通过并支持手动输入数字微调。
  - **圆柱模式**：必须执行 UV 贴图滚动算法 (Cylindrical UV Mapping)，模拟圆柱转动，并非纯平面位移。
  - 目前仅区分平面图和立体图形。
- **光影跨界与连贯性算法 (Seamless Lighting & 3D Shading)**：
  - **三大调子基础光照 (Base Shading)**：方盒不能是纯粹的无光材质 (Unlit)。系统构建基础的环境光 (Ambient) 与定向光 (Directional) 体系。**光照效果不宜过于夸张，参考原版 index.html 的光影叠加层程度**，保持含蓄而有力的存在感。
  - **越界高光顺滑 (Specular Seamless)**：无论是横向（左滑到右）还是纵向（俯视到正视），特殊工艺"流光"不应局限于单面计算。在基础三调子之上，系统会通过计算世界坐标系的方向向量（或全局映射 UV）来触发灰度图的高光反馈，确保光束在转折棱角处顺滑流转，没有断层感。

### 3.2 交互与设备兼容性

- **陀螺仪零点校准**：在用户点击"开始预览"的瞬间，程序必须记录当前设备的 XYZ 角度作为初始基准（Zero Point）。后续的所有光影和视角偏移，均基于此零点计算，防止模型初始状态歪斜。
- **HTTPS 强制**：部署环境必须配置 HTTPS，否则 iOS/Android 现代浏览器将拦截 DeviceOrientation (陀螺仪) API 请求。（注：目前测试用的网页及证书已准备就绪）。
- **微信内置浏览器兼容性与降级 (WeChat WebView Fallback)**：
  - 中国 B2B 场景下，客户 99% 会在微信内直接点开链接。微信内置浏览器（尤其 Android 端）的 WebGL 支持极其有限。
  - **强制降级策略 (Graceful Degradation)**：前端启动时必须执行 `WebGLRenderingContext` 能力检测。一旦 WebGL 不可用或性能不达标，自动降级为"高清多角度静态图片轮播"（后台可配置预渲染截图），并附带提示 `[在浏览器中打开以获得完整 3D 体验]`。
- **社交分享卡片 (OG Meta / 微信分享)**：
  - 裸 URL 分享到微信时必须显示精致的预览卡片（项目名 + 缩略图 + 描述），而非灰色裸链接。
  - 后端根据项目 ID 动态注入 Open Graph meta 标签（`og:title`, `og:image`, `og:description`）。后期可集成微信 JSSDK 控制分享卡片内容。

### 3.3 后端与存储架构 (告别本地堆积)

- **数据库升级**：废弃 db.json。必须引入支持并发的关系型或文档型数据库（如 SQLite 作为轻量级过渡，或直接上 PostgreSQL/MySQL）。
- **数据库定时备份 (DB Backup)**：图片在 R2 上有多副本保护，但数据库（项目配置、版本快照、用户信息）是整套系统的大脑。后端必须配置定时自动备份任务（cron），将数据库快照安全备份至 R2 或其他异地存储。
- **图片存储 (云对象存储)**：
  - 后台接收到图片后，严禁直接堆积在服务器本地 `/uploads` 文件夹。
  - 推荐对接**阿里云 OSS** 或 **Cloudflare R2**。数据库仅保存图片的 URL 链接。
- **设备自适应贴图分发 (Adaptive Asset Delivery)**：
  - **移动端 / 低 DPR**：拉取 150ppi 轻量版，节约带宽，加速加载。
  - **PC 大屏 / 高 DPR**：拉取 300ppi 高清版，充分利用屏幕精度。
  - **CDN 更新缓存刺透 (Cache Busting)**：前端拉取 OSS 图片资源时，URL 后缀必须强行携带当前版本的 Hash 戳（如 `?v=a1b2c3`），彻底解决同名旧图片被 CDN 节点死板缓存导致展示错乱的毁灭级线上 Bug。
- **资源自动处理**：
  - 后台集成图片处理中间件（如 sharp），上传时自动将原图转换为 300ppi 高清和 150ppi 小尺寸的 WebP 格式，并生成列表页缩略图。参考苹果网页使用分层加载模式。
  - **基于相对位置窗口的强制重命名与绑定机制**：彻底摒弃用户原文件名（如 `微信截图_xyz.jpg`）。当用户在后台的“正视图-底图”窗口上传图片时，系统在写入 OSS 时必须强制重命名为标准英文结构（例：`[项目ID]_front_base.webp`）。上传位置（相对位置）直接决定物理文件名，确保前端 3D 拼接时 100% 盲拿不错乱。

### 3.4 安全与防御机制

- **防爆破设计 (递增式冻结)**：针对 6 位提取码接口，必须实现速率限制（Rate Limiting）。同一 IP 连续输错 5 次冻结 15 分钟，之后每次错误冻结时间递增（如 30 分钟、1 小时...逐步拉长），杜绝暴力穷举。
- **访问鉴权**：后端接口需采用 JWT (JSON Web Token) 鉴权，确保客户只能拉取自己提取码对应的项目数据。
- **前端资产强制防盗（双保险防拖拽）**：
  - 吸取旧版教训：单独依赖 CSS（`user-select: none`, `-webkit-user-drag: none`）无法拦截所有用户行为。
  - 新版本必须增加 JavaScript 级别的强制事件拦截：禁止右键菜单 (`contextmenu.preventDefault()`) 且禁止长按拖拽取图 (`dragstart.preventDefault()`)。
- **Sentry 监控与致命级阻断 (Error Reporting & Blocking)**：系统需全量接入 Sentry。当 WebGL 渲染发生致命异常时（而非静默黑屏），不仅向工程师发送堆栈报告，前端必须通过红字横幅向客户友好示警（例如：“图形芯片遇阻，已切换至备用渲染模式”），保证最底线的知情权。
- **隐形水印追踪 (可选防泄漏)**：若有极高保密需求，生成的每一帧 WebGL 画面内核可叠加与访问者 IP/提取码 挂钩的不可见数字水印，截屏外泄后可反向溯源。

## 4. 开发阶段划分建议

请按照以下阶段逐步交付代码：

### Phase 1: 核心渲染引擎攻坚 (前端独立页)

- **任务**：先不写后台。硬编码注入假数据。测试图片在 `public/project_A/v1` 中，所有中文文件名统一按英文逻辑重命名（如 `front_base.png`、`left_specular.png`、`top_base.png` 等）。正视图/背视图共用灰度图，左视图/右视图共用灰度图，俯视图/仰视图共用灰度图，共 6 面。
- **产出**：完成 Three.js/Pixi.js 的 Shader 编写，实现方盒/圆柱两种模式的蒙版、灰度高光计算及陀螺仪接入。

### Phase 2: 数据库与存储底座 (后端基础)

- **任务**：搭建 Node.js/Express 服务，接入 SQLite/PostgreSQL，集成图片转 WebP 及云存储上传逻辑，实现上传后自动命名与自动链接绑定。

### Phase 3: Web 管理后台开发 (Admin)

- **任务**：实现管理员登录、项目 CRUD、图层配置（深度参数调节）、草稿增量编辑与正式发布状态切换。草稿模式需支持对已有配置的局部修改（如替换单面图片、调整属性），无需重建。

### Phase 4: 门户与客户校验流 (Home & Gallery)

- **任务**：实现双入口 UI、提取码递增式防爆破登录、动态生成该客户的专属项目列表，并与 Phase 1 的渲染引擎打通。
